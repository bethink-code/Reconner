## 🔧 Index Implementation

### Step 1: Composite Index for Period + Match Status (PRIMARY)

This is the most important index - it speeds up almost every query.

```sql
-- Index for: period + match status + source type queries
CREATE INDEX IF NOT EXISTS idx_transactions_period_match_source 
  ON transactions(period_id, match_status, source_type);

-- Why this works:
-- 1. period_id first = narrows down to relevant period
-- 2. match_status second = enables "WHERE match_status = 'unmatched'" filters
-- 3. source_type third = enables "WHERE source_type = 'bank'" filters

-- Queries that benefit:
-- - Summary aggregations (COUNT by match_status)
-- - Filtered transaction lists (unmatched bank, matched fuel, etc.)
-- - Match rate calculations
```

**Expected Impact:**
- Summary queries: 1s → 200-300ms
- Filtered queries: 150ms → 30-50ms

---

### Step 2: Index for Date Sorting (SECONDARY)

Speeds up date-based sorting and range queries.

```sql
-- Index for: date sorting and filtering
CREATE INDEX IF NOT EXISTS idx_transactions_period_date 
  ON transactions(period_id, transaction_date DESC, transaction_time DESC);

-- Why DESC?
-- Most queries sort newest-first: ORDER BY transaction_date DESC
-- Index matches query sort order = no additional sorting needed

-- Queries that benefit:
-- - Paginated lists with date sorting
-- - "Recent transactions" views
-- - Date range filters
```

**Expected Impact:**
- Date-sorted queries: 150ms → 30-50ms
- Recent transactions: 200ms → 40ms

---

### Step 3: Partial Index for Card Transactions (SPECIALIZED)

Speeds up card-only queries without indexing cash/unknown.

```sql
-- Partial index: only indexes card transactions
CREATE INDEX IF NOT EXISTS idx_transactions_card_only 
  ON transactions(period_id, source_type, transaction_date DESC)
  WHERE is_card_transaction = 'yes';

-- Why partial?
-- - Only 50% of fuel transactions are cards
-- - Matching only cares about cards
-- - Smaller index = faster lookups, less storage

-- Queries that benefit:
-- - Finding matchable fuel transactions
-- - Card-only reports
-- - Match candidate searches
```

**Expected Impact:**
- Card transaction queries: 150ms → 20-30ms
- Auto-match candidate search: 500ms → 100ms

---

### Step 4: Index for Amount-Based Matching (MATCHING)

Speeds up finding transactions within amount tolerance.

```sql
-- Index for: amount-based searches
CREATE INDEX IF NOT EXISTS idx_transactions_period_amount 
  ON transactions(period_id, amount, transaction_date);

-- Why this helps:
-- Auto-match searches: "Find fuel transactions within ±R0.10 of R100.00"
-- Without index: Full scan of all fuel transactions
-- With index: Range scan on amount column

-- Queries that benefit:
-- - Auto-match candidate finding
-- - Duplicate transaction detection
-- - Amount discrepancy analysis
```

**Expected Impact:**
- Auto-match speed: 2-5 minutes → 30-60 seconds (for large datasets)

---

### Step 5: Index for Reference Number Lookups (OPTIONAL)

Speeds up invoice/reference searches.

```sql
-- Index for: reference number searches
CREATE INDEX IF NOT EXISTS idx_transactions_reference 
  ON transactions(period_id, reference_number)
  WHERE reference_number IS NOT NULL;

-- Why partial?
-- - Not all transactions have reference numbers
-- - Saves space by not indexing NULLs

-- Queries that benefit:
-- - Invoice grouping
-- - Reference number searches
-- - Duplicate detection
```

**Expected Impact:**
- Invoice grouping: 300ms → 50ms
- Reference lookup: 100ms → 10ms

---

### Step 6: Indexes for Matches Table

Speeds up match-related queries.

```sql
-- Index for: match lookups by period
CREATE INDEX IF NOT EXISTS idx_matches_period 
  ON matches(period_id, match_type, confidence DESC);

-- Index for: finding transactions by match
CREATE INDEX IF NOT EXISTS idx_transactions_match_id 
  ON transactions(match_id)
  WHERE match_id IS NOT NULL;

-- Queries that benefit:
-- - Match statistics (count, confidence avg)
-- - Finding all transactions in a match
-- - Match quality analysis
```

**Expected Impact:**
- Match stats: 200ms → 30ms
- Match detail lookups: 50ms → 5ms

## 🚀 Deployment Script

### Full Deployment (Run Once)

```sql
-- ============================================
-- RECONCILIATION SYSTEM - PERFORMANCE INDEXES
-- ============================================
-- Run time: ~2-5 minutes (depends on data size)
-- Safe to run: Indexes are created IF NOT EXISTS
-- Can be run multiple times safely

-- Start transaction (optional - for safety)
BEGIN;

-- ============================================
-- PRIMARY INDEXES (Most Important)
-- ============================================

-- 1. Period + Match Status + Source Type
-- Speeds up: Summary aggregations, filtered lists
CREATE INDEX IF NOT EXISTS idx_transactions_period_match_source 
  ON transactions(period_id, match_status, source_type);

-- 2. Period + Date Sorting
-- Speeds up: Date-sorted lists, recent transactions
CREATE INDEX IF NOT EXISTS idx_transactions_period_date 
  ON transactions(period_id, transaction_date DESC, transaction_time DESC);

-- ============================================
-- SPECIALIZED INDEXES
-- ============================================

-- 3. Card Transactions Only (Partial)
-- Speeds up: Matching, card-only reports
CREATE INDEX IF NOT EXISTS idx_transactions_card_only 
  ON transactions(period_id, source_type, transaction_date DESC)
  WHERE is_card_transaction = 'yes';

-- 4. Amount-Based Searches
-- Speeds up: Auto-match candidate finding
CREATE INDEX IF NOT EXISTS idx_transactions_period_amount 
  ON transactions(period_id, amount, transaction_date);

-- 5. Reference Number Lookups (Partial)
-- Speeds up: Invoice grouping, reference searches
CREATE INDEX IF NOT EXISTS idx_transactions_reference 
  ON transactions(period_id, reference_number)
  WHERE reference_number IS NOT NULL;

-- ============================================
-- MATCH TABLE INDEXES
-- ============================================

-- 6. Match Period Lookups
-- Speeds up: Match statistics, quality analysis
CREATE INDEX IF NOT EXISTS idx_matches_period 
  ON matches(period_id, match_type, confidence DESC);

-- 7. Transaction Match Lookups
-- Speeds up: Finding transactions by match
CREATE INDEX IF NOT EXISTS idx_transactions_match_id 
  ON transactions(match_id)
  WHERE match_id IS NOT NULL;

-- ============================================
-- ANALYZE TABLES
-- ============================================
-- Update query planner statistics for optimal query plans

ANALYZE transactions;
ANALYZE matches;
ANALYZE reconciliation_periods;
ANALYZE uploaded_files;

-- Commit transaction
COMMIT;

-- ============================================
-- VERIFY INDEXES CREATED
-- ============================================

SELECT 
  schemaname,
  tablename,
  indexname,
  indexdef
FROM pg_indexes
WHERE tablename IN ('transactions', 'matches')
ORDER BY tablename, indexname;
```

## 🔍 Monitoring & Maintenance

### Query 1: Index Size Growth

```sql
-- Monitor index sizes over time
SELECT 
  tablename,
  indexname,
  pg_size_pretty(pg_relation_size(indexname::regclass)) AS index_size,
  pg_size_pretty(pg_relation_size(tablename::regclass)) AS table_size,
  round(100.0 * pg_relation_size(indexname::regclass) / 
        pg_relation_size(tablename::regclass), 1) AS index_to_table_ratio
FROM pg_indexes
WHERE schemaname = 'public'
  AND tablename = 'transactions'
ORDER BY pg_relation_size(indexname::regclass) DESC;
```

**Watch for:**
- Total index size > 2x table size = might have too many indexes
- Individual index > table size = investigate if needed

---

### Query 2: Unused Indexes

```sql
-- Find indexes that are never used (after 1 week in production)
SELECT 
  schemaname,
  tablename,
  indexname,
  pg_size_pretty(pg_relation_size(indexname::regclass)) AS index_size,
  idx_scan as times_used
FROM pg_stat_user_indexes
WHERE schemaname = 'public'
  AND tablename = 'transactions'
  AND idx_scan = 0
  AND indexname NOT LIKE '%_pkey'
ORDER BY pg_relation_size(indexname::regclass) DESC;
```

**Action:** If an index shows 0 usage after 1 week, consider dropping it.

---

### Query 3: Index Bloat Check

```sql
-- Check for bloated indexes (needs maintenance)
SELECT 
  schemaname,
  tablename,
  indexname,
  pg_size_pretty(pg_relation_size(indexname::regclass)) AS index_size,
  round(100.0 * pg_relation_size(indexname::regclass) / 
        NULLIF(pg_relation_size(tablename::regclass), 0), 1) AS bloat_ratio
FROM pg_indexes
WHERE schemaname = 'public'
  AND tablename = 'transactions'
ORDER BY pg_relation_size(indexname::regclass) DESC;
```

**Action:** If bloat_ratio > 50%, consider REINDEX:
```sql
REINDEX INDEX CONCURRENTLY idx_transactions_period_match_source;
```

## ⚠️ Important Notes

### Note 1: Index Maintenance
Indexes need periodic maintenance:

```sql
-- Rebuild indexes if they become bloated
REINDEX TABLE CONCURRENTLY transactions;

-- Update statistics after large data changes
ANALYZE transactions;
```

**Schedule:** Monthly or after bulk imports

---

### Note 2: Index Cost
- **Storage:** Indexes add ~40-50% to table size
- **Write speed:** Inserts/updates slightly slower (5-10%)
- **Read speed:** Queries 70-80% faster

**Trade-off:** Worth it - reads vastly outnumber writes in this app

---

### Note 3: Query Planner
PostgreSQL automatically chooses whether to use indexes:

```sql
-- Force planner to show its decision
EXPLAIN (ANALYZE, BUFFERS, TIMING) 
SELECT * FROM transactions WHERE period_id = '...';
```

**Look for:**
- `Index Scan` = Good (using index)
- `Seq Scan` = Bad (full table scan)
- `Bitmap Heap Scan` = OK (partial index use)

## 🔧 Troubleshooting

### Problem 1: Index Not Being Used

**Symptom:** `EXPLAIN` shows `Seq Scan` instead of `Index Scan`

**Causes:**
1. Table too small (PostgreSQL thinks full scan is faster)
2. Query returns most of table (index overhead not worth it)
3. Statistics out of date

**Fix:**
```sql
-- Update statistics
ANALYZE transactions;

-- Force index (testing only - don't use in production)
SET enable_seqscan = off;
EXPLAIN SELECT ...;
SET enable_seqscan = on;
```

---

### Problem 2: Query Slower After Indexes

**Symptom:** Queries slower than before indexing

**Causes:**
1. Wrong index used by planner
2. Index bloated
3. Too many indexes (planner confused)

**Fix:**
```sql
-- Check which index is being used
EXPLAIN (ANALYZE, BUFFERS) 
SELECT * FROM transactions WHERE ...;

-- If wrong index chosen, update statistics
ANALYZE transactions;

-- Or increase statistics target
ALTER TABLE transactions 
ALTER COLUMN period_id SET STATISTICS 1000;
ANALYZE transactions;
```

---

### Problem 3: High Disk Usage

**Symptom:** Database size increased significantly

**Expected:** Indexes add ~40-50% to table size
- transactions table: 15 MB
- All indexes: ~15-20 MB total
- Total increase: ~20 MB

**If much higher:**
```sql
-- Check index sizes
SELECT 
  pg_size_pretty(pg_total_relation_size('transactions')) AS total_size,
  pg_size_pretty(pg_relation_size('transactions')) AS table_size,
  pg_size_pretty(pg_indexes_size('transactions')) AS indexes_size;

-- Drop unused indexes
DROP INDEX IF EXISTS idx_name_of_unused_index;
```

## 🎯 Rollback Plan

If indexes cause problems:

```sql
-- Drop all new indexes
DROP INDEX IF EXISTS idx_transactions_period_match_source;
DROP INDEX IF EXISTS idx_transactions_period_date;
DROP INDEX IF EXISTS idx_transactions_card_only;
DROP INDEX IF EXISTS idx_transactions_period_amount;
DROP INDEX IF EXISTS idx_transactions_reference;
DROP INDEX IF EXISTS idx_matches_period;
DROP INDEX IF EXISTS idx_transactions_match_id;

-- Refresh statistics
ANALYZE transactions;
ANALYZE matches;
```

**Impact of rollback:**
- Queries return to pre-index speed
- No data loss
- No functional changes
