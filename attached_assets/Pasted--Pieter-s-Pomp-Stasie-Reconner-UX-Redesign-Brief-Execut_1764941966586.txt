# Pieter's Pomp Stasie Reconner
## UX Redesign Brief

---

# Executive Summary

The matching engine now works well (75% match rate). The UX needs to catch up.

**Core Problem:** The UI is built around fuel data as the focus, but users need to reconcile BANK transactions.

**The Question Users Are Asking:** "Did all my bank transactions match to fuel records? Which ones didn't? Why?"

---

# Mental Model Shift

```
OLD: "Here's your fuel data. How much can we verify with bank statements?"

NEW: "Here's your bank transactions. Which ones matched? Which need investigation?"
```

---

# Three Payment Types = Three Workflows

The fuel system has three payment types with completely different reconciliation needs:

| Type | Volume | Reconciliation Approach |
|------|--------|------------------------|
| **Card** | 12,138 txns (R6.5M) | Match each bank transaction to fuel sale |
| **Cash** | 12,552 txns (R1.9M) | Verify daily/weekly deposit totals |
| **Debtor** | 678 txns (R517K) | Track accounts, match to monthly payments |

**Current UI:** Treats everything the same
**Required:** Three distinct sections/workflows

---

# Redesigned Flow

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: Fuel Data       Step 2: Bank Data      Step 3     │
```

### Improvements:
- Add **Matching Rules as Step 3** (before reconciliation runs)
- **Auto-run matching** after wizard completes
- Show **coverage warning** if bank data is sparse
- **Sequential file upload** - one file at a time with clear context
